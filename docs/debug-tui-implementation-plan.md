# Debug TUI Implementation Plan

Updated: 2026-07-02.

This document captures the implementation plan for a terminal UI that lets an
operator talk to Chudbot through a local debug platform. It replaces the earlier
web-debug idea with a TUI. It is a plan only. Do not treat it as already
implemented.

## Goal

Build a local terminal UI for debugging and testing Chudbot as a real platform
surface:

- Send messages to Chudbot without Discord.
- Watch model output and runtime activity live.
- Inspect conversation ids, turn status, tool calls, usage, and trace state.
- Exercise retry, stop, resume, agent selection, media attachment, and memory
  behavior through the same bot runtime paths used by Discord.
- Keep the debug interface local and operator-facing, not a public chat
  surface.

## Non-Goals

- Do not replace the existing web trace viewer.
- Do not add public unauthenticated debug APIs.
- Do not build a Discord clone in terminal form.
- Do not bypass `BotRuntime` or write a second turn runner.
- Do not require Discord credentials for TUI mode.
- Do not persist per-token deltas in the first pass.

## Recommended Crate Layout

Add two focused pieces:

```text
crates/chudbot-debug-platform
  -> synthetic platform implementation and in-memory timeline

crates/chudbot-tui
  -> terminal UI, input handling, layout, and debug command handling

crates/chudbot-bin
  -> new `tui` command that wires config, storage, providers, media, bot
     runtime, debug platform, and TUI supervisor together
```

`chudbot-debug-platform` should not depend on `chudbot-tui`. The platform can
be tested headlessly and reused by future integration tests.

Use `ratatui` plus `crossterm` for the first TUI pass. Ratatui's own docs
describe it as a lightweight Rust TUI library with widgets and utilities, and
its current docs recommend using the main `ratatui` crate for applications.
They also document Crossterm as the default backend for most platforms.

## Runtime Shape

TUI mode runs three in-process services:

- `BotRuntime`, built from the normal config and provider registries.
- `DebugPlatform`, implementing `MessagePlatform` and
  `MessagePlatformEvents`.
- `TuiApp`, rendering the debug platform timeline and sending user actions
  into the debug platform handle.

The data flow should be:

```text
keyboard input
  -> TUI command/input reducer
  -> DebugPlatformHandle::post_user_message
  -> PlatformEvent::MessageCreated
  -> BotRuntime::handle_event
  -> platform turn delivery / send_message calls
  -> DebugPlatform timeline events
  -> TUI redraw
```

The bot should see the debug TUI as just another platform. The TUI should not
call private bot turn methods.

## Synthetic Platform Identity

Use stable, explicit platform refs:

- Platform name: default `debug-tui`, configurable by CLI flag.
- Workspace id: one generated id per TUI session by default.
- Channel id: `main` for the first pass.
- User id: configurable, default `operator`.
- Bot id: `chudbot`.
- Message ids: monotonic or UUID-backed ids generated by the debug platform.

Every submitted TUI message should be admitted as addressing the bot. After the
platform-boundary refactor, this should be expressed through the debug
platform's admission method, not by injecting Discord-style mention markup.

For isolation, each TUI session should default to its own workspace id. Add a
CLI flag later for joining a named workspace when we intentionally want shared
memory or history across TUI sessions.

## Debug Platform Responsibilities

`chudbot-debug-platform` owns a process-local timeline:

- User messages.
- Assistant messages.
- Provisional streamed output.
- Status updates.
- Tool started/finished events.
- Reactions or semantic controls.
- Deleted messages.
- Attachments.
- Typing/activity state.

It should implement the platform contract this way:

- `bot_user`: return the synthetic Chudbot user.
- `register_commands`: store definitions for inspection or no-op for MVP.
- `next_event`: receive queued `PlatformEvent`s from the TUI handle.
- `send_message`: append an assistant/status/error message to the timeline and
  return `PostedMessage`.
- `delete_message`: mark a timeline message deleted.
- `add_reaction` / `remove_own_reaction`: convert to timeline control/status
  state, not necessarily emoji UI.
- `typing`: set transient activity state.
- `fetch_messages`: return visible timeline messages in the current debug
  workspace/channel.
- `message_context`: render `type = "debug_tui_message"` JSON with author,
  workspace, channel, message, content, mentions, and attachments.
- `parent_channel`: return the channel itself for MVP.

After the platform-boundary refactor, prefer the semantic turn-delivery methods
over raw `send_message` for normal turn output.

## TUI Experience

The first screen should be the working debugger, not a landing page.

Recommended layout:

- Main transcript pane:
  - user messages;
  - streamed/provisional assistant output;
  - final assistant output;
  - status updates;
  - tool-call summaries.
- Input pane:
  - multiline text entry;
  - send/cancel affordance;
  - command mode for slash-like debug commands.
- Right inspector pane:
  - active agent/provider/model;
  - conversation id and turn id;
  - turn status;
  - usage totals;
  - active tool calls;
  - stopped/running state.
- Bottom status bar:
  - connection/runtime status;
  - current workspace/channel/user;
  - key hints.

Initial commands:

- `/new`: start a new debug workspace/channel.
- `/agent list`: show configured agents.
- `/agent show`: show active selection.
- `/agent set <name>`: set platform or session selection.
- `/agent clear`: clear the session selection.
- `/retry`: retry the latest failed turn.
- `/stop`: stop the active conversation and cancel in-flight work.
- `/resume`: resume a stopped conversation.
- `/trace`: toggle trace inspector.
- `/help`: show command list.

Keep the command layer local to the TUI unless it is testing platform command
behavior. For platform command behavior, enqueue real `PlatformEvent::Command`
events and let `chudbot-bot` handle `/chudbot-agent`.

## Streaming Behavior

The platform-boundary refactor should let the TUI consume semantic turn events.
The TUI should render them as follows:

- `TextDelta`: append to a provisional assistant block.
- `ReasoningDelta`: show in the inspector, collapsed by default.
- `ToolStarted`: show active tool state.
- `ToolFinished`: move the tool into the turn history.
- `StatusText`: render as an assistant status message.
- `finish_turn`: reconcile provisional text with the final stored assistant
  content.

Important: model text deltas are provisional. A provider step can stream text
and still finish as a client-tool step, so the UI must treat the final turn
outcome as authoritative.

## Attachments and Media

First pass:

- Allow pasting or typing a local path command such as `/attach <path>`.
- The TUI reads bytes locally and stores them behind a debug-platform attachment
  URL that `chudbot-bot` can fetch through its existing attachment download
  path.
- The `PlatformMessage` attachment should include filename, MIME type when
  known, size, and URL.

Later:

- Drag/drop is out of scope for a terminal app.
- Add media preview metadata in the inspector.
- Add a command to attach existing `media://...` assets to a new user message
  for replay testing.

## Configuration

Add a new command:

```sh
cargo run -p chudbot-bin -- tui
```

TUI mode should:

- Load the normal config.
- Run migrations only when explicitly asked, matching existing command
  semantics.
- Build storage, providers, media, and event bus normally.
- Build a debug platform even if `[platforms]` has no Discord entry.
- Bind the debug platform to `[bot].default_agent` unless a
  `[bot.platforms.debug-tui]` binding exists.

Possible future TOML:

```toml
[platforms.debug-tui]
kind = "debug_tui"

[bot.platforms.debug-tui]
agent = "chudbot"
```

Do not require operators to add this config for the first usable TUI. The CLI
can synthesize the platform binding in memory.

## Implementation Phases

### Phase 1: Headless Debug Platform

- Add `chudbot-debug-platform`.
- Implement `MessagePlatform` and a cloneable `DebugPlatformHandle`.
- Add unit tests for posting user messages, assistant sends, reactions,
  deletion, fetch history, and message context rendering.
- Add a small integration test with a fake LLM provider and real `BotRuntime`.

### Phase 2: TUI Shell

- Add `chudbot-tui`.
- Build terminal setup/restore, app state, event loop, and transcript rendering.
- Use a fake local timeline first, before wiring the full bot.
- Add reducer tests for keyboard commands and timeline events.

### Phase 3: Runtime Wiring

- Add `chudbot-bin tui`.
- Reuse service construction from `serve` where practical.
- Start `BotRuntime` with debug platform events and the TUI as sibling tasks.
- Make shutdown restore the terminal even on bot/runtime error.

### Phase 4: Basic Chat

- Send plain text from TUI to Chudbot.
- Show assistant replies after final delivery.
- Show working/success/error/refused state.
- Display the conversation UUID and trace URL.

### Phase 5: Streaming

- Depend on the platform-boundary refactor's semantic turn events.
- Render provisional token output.
- Reconcile final assistant content after completion.
- Show tool activity in the inspector.

### Phase 6: Controls

- Implement `/retry`, `/stop`, and `/resume`.
- Map UI actions to debug platform semantic controls.
- Verify retry goes through normal storage `prepare_retry`.
- Verify stop cancels in-flight turns through runtime cancellation.

### Phase 7: Trace Inspector

- Subscribe to the existing `EventBus`.
- Reload `ConversationSnapshot` on conversation events.
- Render context items, model steps, tool traces, usage, and replay assets in
  the inspector pane.
- Keep the web trace viewer as the richer external view.

### Phase 8: Attachments

- Add `/attach <path>` for image and audio testing.
- Serve debug attachment URLs from the debug platform or expose a local in-memory
  fetch path the bot HTTP client can access.
- Verify image/audio attachment persistence and transcription behavior.

## Testing Plan

- Unit test the debug platform timeline independently of the TUI.
- Unit test TUI reducers without a terminal backend.
- Integration test `BotRuntime` plus debug platform with a fake streaming LLM.
- Integration test retry, stop, and failed-turn cleanup.
- Test terminal restore paths by isolating setup/restore behind a small wrapper.
- Keep external provider and Discord calls mocked.

## Acceptance Criteria

- `cargo run -p chudbot-bin -- tui` opens a terminal chat surface.
- A user can send a message and receive a Chudbot answer without Discord.
- The answer follows the same storage, trace, agent, tool, and media paths as
  Discord turns.
- The TUI shows live turn status and final trace metadata.
- Retry and stop work through normal bot semantics.
- TUI mode works when no Discord platform token is configured.

## Open Questions

- Should TUI command history persist across runs?
- Should the default debug workspace be ephemeral or named by config?
- Should TUI mode expose real platform commands or keep local commands separate?
- Should local attachments be served over an internal HTTP listener, or should
  the bot media-download path accept a debug-platform media ref directly?
- Should the TUI have a read-only trace-inspection mode for existing
  conversations?
