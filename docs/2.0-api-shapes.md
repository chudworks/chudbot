# Chudbot 2.0 Migration Handoff

Updated: 2026-06-15.

The first pass of Chudbot 2.0 is now the working tree target. The legacy
`grok-discord-bot-core` and `grok-discord-bot-bin` crates were removed after
the split; the `chudbot-*` crates are the source of truth.

## Current Crates

- `chudbot-api`: shared contracts for ids, transcripts, agent loop, tools,
  media, platform events, storage, live events, retry, and usage.
- `chudbot-bot`: platform-neutral bot orchestration.
- `chudbot-discord`: Twilight message platform implementation.
- `chudbot-web`: Axum JSON API, SSE, media serving, static SPA serving, and
  crawler controls.
- `chudbot-storage-sqlx`: Postgres `BotStorage` implementation with embedded
  root migrations.
- `chudbot-asset-local`: local filesystem `MediaStore`.
- `chudbot-xai`: xAI LLM/image/video/audio provider.
- `chudbot-gemini`: Google Gemini LLM, Nano Banana image, and Veo video
  provider.
- `chudbot-openai`: OpenAI Responses LLM and image provider.
- `chudbot-openai-compat`: OpenAI-compatible Chat Completions LLM provider
  for local/model-gateway hosts such as vLLM.
- `chudbot-anthropic`: Anthropic Messages LLM provider.
- `chudbot-bin`: TOML config loader, service registry builder, tracing setup,
  migration runner, and process supervisor.

  `[image.<name>.pricing.<model>]`. Anthropic estimates account for uncached

- Anthropic LLM server tools persist raw Messages content as provider
  continuation data, so hosted web search can resume `pause_turn` responses and
  carry search/citation state across replayed turns.
- Added a v2 web read DTO with a `users` map containing display names and
  cached avatar media URIs for frontend rendering.
- Rendered turn replay assets in the React viewer so generated image/video
  media persisted in `turn_assets` is visible even when it is not present in
  live tool trace payloads.
- Preserved xAI Responses manual-state replay: provider continuations live in
  ordered `turn_attempt_model_steps` rows, rebuild into model transcripts,
  replay provider output with malformed encrypted reasoning blobs omitted, use
  the conversation UUID as `prompt_cache_key` and `x-grok-conv-id`, add stable

  state out of the public turn DTO.
- Added the first-pass user memory system: API storage contracts, Postgres
  ledger/profile/job tables, bot-owned memory tools and prompt guidance,
  lookup-only profile retrieval, and an in-process diary/compaction scheduler
  driven by `[memory]`. Background diary and compaction workers resolve normal
  `[bot.agents.memory_diary]` and `[bot.agents.memory_compact]` configs, with
  built-in defaults synthesized only when those agent entries are omitted.
  Memory diary entries and compact profile revisions store their
  background-agent provider, model, and usage payloads for auditability.
  Diary scheduling keeps non-empty source windows on the configured cadence
  anchored to the previous diary window end with half-open source windows,
  while source-empty intervals may be skipped. Terminal diary jobs also advance
  the diary window cursor, so exhausted failed windows are skipped instead of
  recreated forever. Compact memory profiles are loaded from the latest
  `user_memory_document_versions` row; there is no separate current-document
  table.
- Added first-pass incoming voice-message support: Discord voice/audio
  attachments are stored as `file://audio/...` media, platform message JSON gets
  `audio_attachments` plus per-attachment `audio_uri`, agents can call
  `transcribe_audio`, xAI Speech-to-Text usage is preserved or estimated, and
  the viewer can serve/render stored audio assets.
- Added stored media utility tools: `stat` validates a `file://...` media URI
  and returns existence/MIME/size metadata, `public_url` returns a configured
  public asset URL when available, and `read` exposes only verified stored image
  assets to the next model step. `attach` explicitly queues a verified stored
  image asset for the final platform reply, deduplicating by URI with media
  already queued by generation tools. `read` and `attach` do not return bytes
  and reject videos, audio, PDFs, unknown MIME types, public URLs, and local
  filesystem paths.
- Added no-mention voice-message ingress: when an audio message would otherwise
  be ignored, the bot uses the selected agent's configured audio transcriber
  directly to continue a replied-to conversation or, when
  `audio_transcription.wake_word` is set, detect that spoken wake word for a
  new conversation. Successful preflight transcripts are supplied as plain user
  text and do not expose a model-facing audio URI, avoiding duplicate
  transcription calls. If the selected agent has no transcription binding,
  lacks a wake word for standalone audio, or transcription yields no text, the
  no-mention audio message is ignored.
- Added a memory-diary special case for voice turns: completed-turn memory
  windows load successful `transcribe_audio` client tool results from
  `turn_attempt_tool_traces` and render the transcription text into the diary
  prompt alongside the normal user/assistant text.
- Added memory-job tracing context: claimed diary/compaction jobs carry the
  memory agent name, message provider, scope key, scope id, Discord guild id
  when applicable, subject user id, best-effort subject display name, job id,
  kind, and attempt count so provider debug request logs can be tied back to the
  memory subject.

## API Shape

Dependency direction remains:

```text
chudbot-bin
  -> chudbot-bot
  -> chudbot-web
  -> chudbot-discord
  -> chudbot-storage-sqlx
  -> chudbot-asset-local
  -> chudbot-xai / chudbot-gemini / chudbot-openai
  -> chudbot-openai-compat / chudbot-anthropic

chudbot-* implementation crates -> chudbot-api
```

`chudbot-api` must stay free of Twilight, SQLx, Reqwest, Axum, and concrete
provider config.

The bot config is agent-first:

- `bot.agents`: prompt, provider name, `ModelSpec`, optional server/client tool
  restrictions, loop limits, media generation/transcription bindings, and
  subagent bindings.
  `ModelSpec.provider_options.value` is decoded by the concrete LLM kind behind
  the agent provider key, so the config does not repeat the provider kind there.
  The OpenAI Responses provider accepts `reasoning_effort` plus optional
  `reasoning_summary = "auto" | "concise" | "detailed"`; summaries are opt-in
  and are what the viewer can expose as safe reasoning text. It also accepts
  `text_verbosity = "low" | "medium" | "high"`, mapped to OpenAI
  `text.verbosity`.
  The Gemini provider accepts `thinking_level`, `thinking_budget`,
  `include_thoughts`, and `extra_body`; `web_search` maps to Gemini Google
  Search grounding.
  The Anthropic Messages provider accepts `thinking`, passed through as the
  request's thinking object, and `effort`; returned `thinking` blocks are
  exposed as viewer-safe reasoning summaries, while Anthropic bills full
  thinking through ordinary output tokens.
- `bot.agents.memory_diary`, `bot.agents.memory_compact`,
  `bot.agents.tos_preflight`, and `bot.agents.conversation_title`: magic system
  agent names used by background memory jobs, moderation preflight, and
  conversation title generation. If a magic system agent entry is omitted, the
  runtime synthesizes the previous built-in default for that missing agent.
  These reserved system agents are hidden from `/chudbot-agent` list/set.
- `llm`, `image`, `video`, `audio`: named provider services.
- `platforms`: named message platform services.
- `bot.platforms`: platform default agent bindings.
- Discord slash commands register globally on startup so every guild where the
  app is installed can use the runtime commands.
- `bot.thread_threshold_chars` and `bot.thread_threshold_lines`: auto-thread
  new-conversation replies when either raw length or approximate visible rows
  would dominate a channel.
- `memory`: global user-memory scheduler settings, including polling, leases,
  source-gated compaction interval, diary backfill window, diary interval, job
  limits, and retry controls.
- `bot.agents.<name>.memory`: per-agent opt-in for the `lookup_user_memory`,
  `remember_user_memory`, and `forget_user_memory` client tools. Compact
  profiles, recent un-compacted events, and the last three diary entries are
  returned by lookup calls, not injected into ordinary turn context. Subagents
  do not receive memory guidance by default.
- Platform message context includes structured `mentioned_users` entries so
  agents can target memory lookups by stable user id instead of parsing raw
  mention text.
- Platform message context includes stored audio references when a message has
  voice/audio attachments. The root `audio_attachments` array and each matching
  attachment's `audio_uri` contain stable `file://audio/...` URIs for the
  `transcribe_audio` client tool. Audio is not replayed as a native model media
  block today; images remain the only current transcript media input.
- `bot.agents.<name>.audio_transcription.wake_word` is optional. When set, it
  enables direct preflight transcription for standalone no-mention audio so the
  transcript can start a conversation by wake word. When omitted, no-mention
  audio outside an existing conversation is ignored. The configured wake word is
  also sent as a default transcription keyterm for xAI STT biasing.
- `bot.agents.<name>.video_generation.rate_limit` optionally caps active video
  generations per platform scope over a rolling interval, with configured
  platform scopes allowed to bypass the cap. Enforcement counts pending
  `video_jobs` rows plus done rows with a stored `output_uri`.
- Memory diary jobs receive successful `transcribe_audio` results for completed
  subject-user turns as explicit audio-transcription snippets. Other client
  tool traces are still excluded from diary input unless added deliberately.
- `logging.filter`: parsed as a tracing `EnvFilter`; invalid filters are
  configuration errors rather than runtime fallbacks. Use
  `info,chudbot=debug` for debug logs from first-party `chudbot-*` crates and
  info logs from dependencies.
- Memory diary/compaction worker spans attach `message_provider`, `scope_key`,
  `scope_id`, `guild_id` when the scope is a Discord guild, and
  `user_id`/`target_user_id`; when `platform_users` has a readable profile name,
  the span also includes `target_user_name`. Downstream provider request spans
  inherit this context.

The web API now serves v2 read models:

- `GET /api/config` -> `{ title_prefix, version }`
- `GET /api/conversations/{uuid}` -> `{ conversation, turns, users }`
  where client tool traces expose model-facing `result` and omit
  `trace_payload` unless the stored trace/debug payload differs. Each turn also
  exposes normalized `reasoning` metadata with safe summary text and aggregated
  model-step reasoning token counts; encrypted/raw provider continuation state
  remains replay-only.
- `GET /api/conversations/{uuid}/events` -> SSE
- `/images/{name}`, `/videos/{name}`, `/audio/{name}`, `/avatars/{name}` ->
  media
- `/assets/*` -> cached Vite bundle assets; extensionless fallbacks -> React SPA
  index, cached only in non-debug builds

`PlatformMessage` keeps provider-neutral ids, author metadata, content,
attachments, and reply references. `MessagePlatform::message_context` owns the
model-facing JSON vocabulary for a platform message, so Discord-specific terms
such as guild, channel, and guild display name stay in `chudbot-discord`.

## Commands

Runtime commands are:

- `/chudbot-privacy in|out|status`
- `/chudbot-mode show|set`
- `/chudbot-agent list|show|set|clear`

The old `/grok-persona` command vocabulary is not carried forward.

Command invocations use `PlatformCommandInput` / `PlatformCommandValue` for
platform-normalized option values. Platform crates may use native SDK types
internally, but they must not leak them into `chudbot-api`.

## Known Remaining Gaps

These were identified in the old-vs-new feature audit and should be treated as
follow-up work, not hidden regressions:

- The viewer has v2 user metadata again, but app-version commit metadata is not
  yet exposed as a dedicated per-conversation read map.

## Verification Bundle

Use this before calling a migration step complete:

```sh
cargo run -p chudbot-bin -- check-config --config config.example.toml
cargo check --all-targets --all-features
cargo test --all-features
cd frontend && bun run typecheck && bun run build
git diff --check
```
