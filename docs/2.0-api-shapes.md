# Chudbot 2.0 Migration Handoff

Updated: 2026-06-04.

The first pass of Chudbot 2.0 is now the working tree target. The legacy
`grok-discord-bot-core` and `grok-discord-bot-bin` crates were removed after
the split; the `chudbot-*` crates are the source of truth.

## Current Crates

- `chudbot-api`: shared contracts for ids, transcripts, agent loop, tools,
  media, platform events, storage, live events, retry, and usage.
- `chudbot-bot`: platform-neutral bot orchestration.
- `chudbot-discord`: Twilight message platform implementation.
- `chudbot-web`: Axum JSON API, SSE, media serving, static SPA serving, and
  crawler controls.
- `chudbot-storage-sqlx`: Postgres `BotStorage` implementation with embedded
  root migrations.
- `chudbot-asset-local`: local filesystem `MediaStore`.
- `chudbot-xai`: xAI LLM/image/video/audio provider.
- `chudbot-openai`: OpenAI Responses LLM and image provider.
- `chudbot-openai-compat`: OpenAI-compatible Chat Completions LLM provider
  for local/model-gateway hosts such as vLLM.
- `chudbot-anthropic`: Anthropic Messages LLM provider.
- `chudbot-bin`: TOML config loader, service registry builder, tracing setup,
  migration runner, and process supervisor.

  lookup-only profile retrieval, and an in-process diary/compaction scheduler
  driven by `[memory]`. Memory diary entries and compact profile revisions store
  their background-agent provider, model, and usage payloads for auditability.
  Diary scheduling keeps non-empty source windows on the configured cadence
  anchored to the previous diary window end with half-open source windows,

  loaded from the latest `user_memory_document_versions` row; there is no

- Added no-mention voice-message ingress: when an audio message would otherwise

  new conversation. Successful preflight transcripts are supplied as plain user
  text and do not expose a model-facing audio URI, avoiding duplicate
  transcription calls. If the selected agent has no transcription binding,
  lacks a wake word for standalone audio, or transcription yields no text, the
  no-mention audio message is ignored.
- Added a memory-diary special case for voice turns: completed-turn memory
  windows load successful `transcribe_audio` client tool results from
  `turn_attempt_tool_traces` and render the transcription text into the diary
  prompt alongside the normal user/assistant text.

## API Shape

Dependency direction remains:

```text
chudbot-bin
  -> chudbot-bot
  -> chudbot-web
  -> chudbot-discord
  -> chudbot-storage-sqlx
  -> chudbot-asset-local
  -> chudbot-xai / chudbot-openai / chudbot-openai-compat / chudbot-anthropic

chudbot-* implementation crates -> chudbot-api
```

`chudbot-api` must stay free of Twilight, SQLx, Reqwest, Axum, and concrete
provider config.

The bot config is agent-first:

- `bot.agents`: prompt, provider name, `ModelSpec`, optional server/client tool
  restrictions, loop limits, media generation/transcription bindings, and
  subagent bindings.
  `ModelSpec.provider_options.value` is decoded by the concrete LLM kind behind
  the agent provider key, so the config does not repeat the provider kind there.
- `llm`, `image`, `video`, `audio`: named provider services.
- `platforms`: named message platform services.
- `bot.platforms`: platform default agent bindings.
- Discord slash commands register globally on startup so every guild where the
  app is installed can use the runtime commands.
- `bot.thread_threshold_chars` and `bot.thread_threshold_lines`: auto-thread
  new-conversation replies when either raw length or approximate visible rows
  would dominate a channel.
- `memory`: global user-memory runtime settings, including provider key,
  polling, leases, source-gated compaction interval, diary backfill window,
  diary interval, job limits, and retry controls.
- `bot.agents.<name>.memory`: per-agent opt-in for the `lookup_user_memory`,
  `remember_user_memory`, and `forget_user_memory` client tools. Compact
  profiles are returned by lookup calls, not injected into ordinary turn
  context. Subagents do not receive memory guidance by default.
- Platform message context includes structured `mentioned_users` entries so
  agents can target memory lookups by stable user id instead of parsing raw
  mention text.
- Platform message context includes stored audio references when a message has
  voice/audio attachments. The root `audio_attachments` array and each matching
  attachment's `audio_uri` contain stable `file://audio/...` URIs for the
  `transcribe_audio` client tool. Audio is not replayed as a native model media
  block today; images remain the only current transcript media input.
- `bot.agents.<name>.audio_transcription.wake_word` is optional. When set, it
  enables direct preflight transcription for standalone no-mention audio so the
  transcript can start a conversation by wake word. When omitted, no-mention
  audio outside an existing conversation is ignored. The configured wake word is
  also sent as a default transcription keyterm for xAI STT biasing.
- `bot.agents.<name>.video_generation.rate_limit` optionally caps successful
  video generations per platform scope over a rolling interval, with configured
  platform scopes allowed to bypass the cap. Enforcement counts existing
  `video_jobs` rows with `status = 'done'` and a stored `output_uri`.
- Memory diary jobs receive successful `transcribe_audio` results for completed
  subject-user turns as explicit audio-transcription snippets. Other client
  tool traces are still excluded from diary input unless added deliberately.
- `logging.filter`: parsed as a tracing `EnvFilter`; invalid filters are
  configuration errors rather than runtime fallbacks. Use
  `info,chudbot=debug` for debug logs from first-party `chudbot-*` crates and
  info logs from dependencies.

The web API now serves v2 read models:

- `GET /api/config` -> `{ title_prefix, version }`
- `GET /api/conversations/{uuid}` -> `{ conversation, turns, users }`
  where client tool traces expose model-facing `result` and omit
  `trace_payload` unless the stored trace/debug payload differs.
- `GET /api/conversations/{uuid}/events` -> SSE
- `/images/{name}`, `/videos/{name}`, `/audio/{name}`, `/avatars/{name}` ->
  media
- `/assets/*` and extensionless fallbacks -> React SPA

`PlatformMessage` keeps provider-neutral ids, author metadata, content,
attachments, and reply references. `MessagePlatform::message_context` owns the
model-facing JSON vocabulary for a platform message, so Discord-specific terms
such as guild, channel, and guild display name stay in `chudbot-discord`.

## Commands

Runtime commands are:

- `/chudbot-privacy in|out|status`
- `/chudbot-mode show|set`
- `/chudbot-agent list|show|set|clear`

The old `/grok-persona` command vocabulary is not carried forward.

Command invocations use `PlatformCommandInput` / `PlatformCommandValue` for
platform-normalized option values. Platform crates may use native SDK types
internally, but they must not leak them into `chudbot-api`.

## Known Remaining Gaps

These were identified in the old-vs-new feature audit and should be treated as
follow-up work, not hidden regressions:

- The viewer has v2 user metadata again, but app-version commit metadata is not
  yet exposed as a dedicated per-conversation read map.

## Verification Bundle

Use this before calling a migration step complete:

```sh
cargo run -p chudbot-bin -- check-config --config config.example.toml
cargo check --all-targets --all-features
cargo test --all-features
cd frontend && bun run typecheck && bun run build
git diff --check
```
